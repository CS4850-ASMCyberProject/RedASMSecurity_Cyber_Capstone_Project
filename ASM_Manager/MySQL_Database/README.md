# 🗄️ ASM_Manager Database – MySQL / MariaDB

🔗 Database Component for ASM Scanning Engine & SOC Pipeline

## 📌 Overview

The ASM Database stores all discovered assets, paths, vulnerabilities, and scan history generated by the ASM Scanning Engine.

It acts as the central data layer connecting:

🌐 ASM Scanning Engine (discovery & scanning)  
🛡️ Wazuh (detection)  
🔄 Shuffle (automation)  
🐝 TheHive (case management)  

## 🎯 Purpose

This database is designed to:

Store discovered assets and services  
Track URL paths and endpoints  
Record vulnerabilities detected by scanners  
Maintain historical scan changes  
Enable structured querying for automation workflows  
Support long-term attack surface analysis

```bash
sudo apt update
```

```bash
sudo mysql_secure_installation
```

Login to MySQL / MariaDB
```bash
sudo mysql -u root -p
```

```bash
  cdn varchar(100) DEFAULT NULL,

);
```

🍹 juice_shop_paths
```bash
CREATE TABLE juice_shop_paths (
  id int(11) NOT NULL AUTO_INCREMENT,
  asset_id int(11) NOT NULL,
  url_path varchar(255) NOT NULL,
  first_seen datetime DEFAULT current_timestamp(),
  PRIMARY KEY (id),
  UNIQUE KEY asset_id (asset_id, url_path),
  CONSTRAINT juice_shop_paths_ibfk_1 
    FOREIGN KEY (asset_id) REFERENCES assets (id) 
    ON DELETE CASCADE
);
```

🌍 url_paths
```bash
CREATE TABLE url_paths (
  id int(11) NOT NULL AUTO_INCREMENT,
  asset_id int(11) NOT NULL,
  url_path varchar(100) DEFAULT NULL,
  status int(11) DEFAULT NULL,
  size int(11) DEFAULT NULL,
  words int(11) DEFAULT NULL,
  line_count int(11) DEFAULT NULL,
  duration int(11) DEFAULT NULL,
  first_seen timestamp NOT NULL DEFAULT current_timestamp(),
  last_changed timestamp NOT NULL DEFAULT current_timestamp() ON UPDATE current_timestamp(),
  PRIMARY KEY (id),
  KEY idx_url_paths_asset_id (asset_id),
  KEY idx_url_paths_status (status),
  CONSTRAINT fk_url_paths_assets 
    FOREIGN KEY (asset_id) REFERENCES assets (id) 
    ON DELETE CASCADE
);
```

🚨 vulnerabilities
```bash
CREATE TABLE vulnerabilities (
  id int(11) NOT NULL AUTO_INCREMENT,
  asset_id int(11) NOT NULL,
  vulnerability_id varchar(255) NOT NULL,
  type varchar(20) DEFAULT NULL,
  severity varchar(20) DEFAULT NULL,
  matched_at text DEFAULT NULL,
  extracted_results longtext CHARACTER SET utf8mb4 COLLATE utf8mb4_bin 
    DEFAULT NULL CHECK (json_valid(extracted_results)),
  vulnerability_score int(11) DEFAULT 0,
  first_seen timestamp NOT NULL DEFAULT current_timestamp(),
  last_changed timestamp NOT NULL DEFAULT current_timestamp() ON UPDATE current_timestamp(),
  PRIMARY KEY (id),
  UNIQUE KEY uniq_asset_vuln (asset_id, vulnerability_id),
  KEY idx_asset_id (asset_id),
  KEY idx_vulnerability_id (vulnerability_id),
  KEY idx_severity (severity),
  CONSTRAINT fk_vulnerabilities_asset 
    FOREIGN KEY (asset_id) REFERENCES assets (id) 
    ON DELETE CASCADE
);
```

📜 assets_scan_history
```bash
CREATE TABLE assets_scan_history (

    ON DELETE CASCADE
);
```

## 🧠 Key Features

🌐 Asset-Centric Design
All tables link back to assets  
Enables full attack surface mapping  

🔄 Historical Tracking  
Dedicated *_scan_history tables  
Tracks:  
Changes in assets  
Path updates
Vulnerability evolution  

🚨 Vulnerability Intelligence
Stores:  
Severity  
Type  
Extracted JSON results  
Supports correlation with SIEM alerts  
⚡ Optimized Querying  

Indexed fields for:  
asset_id  
status  
severity  
vulnerability_id  

🔗 Integration in ASM Pipeline

This database feeds:

🔍 ASM Scanning Engine → writes data  
🛡️ Wazuh → detects attacks on assets  
🔄 Shuffle → queries + automates response  
🐝 TheHive → creates cases based on findings  
🚀 Summary  

The ASM Database provides a structured and scalable backend that:

Stores attack surface data
Tracks changes over time
Supports vulnerability analysis
Enables automation workflows
Bridges offensive scanning with defensive response
