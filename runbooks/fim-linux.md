# Runbook: File Integrity Monitoring (Linux)

**Applies to:**
- Wazuh File Integrity Monitoring alerts
- Alert severity: Hogh or Critical
- MITRE ATT&CK: T1565.001

---

## Purpose
This runbook explains how to respond to file integrity alerts generated by Wazuh on Linux systems.

---

## Trigger Conditions
Use this runbook when:
- A Wazuh FIM alert is generated
- The alert involves a monitored file or directory
- The alert severtiy is high or critical

Common examples:
- /etc/passwd
- /etc/shadow
- /etc/ssh/sshd_config
- Application configure files

---

## Initial Triage (%-10 minutes)

1. Identify the file that was modified
    - Is it a system-critical or application-critical file?

2. Check when the change occurred
    - Was this during scheduled maintenance?
    - Was a change expected?

3. Identify the user or process
    - Who made the change?
    - Was it root, a service account, or unknown?

4. Look for related alerts
    - Authentication failures?
    - New processes?
    - Network activity?

---

## Decision Guide

- If the change is expected and approved
  → Close the alert as benign and document the reason

- If the change is unexpected but low risk
  → Monitor for repeat activity

- If the change is unexpected and affects a sensitive file
  → Escalate immediately

---

## Response Actions
- Preserve relevant logs and file hashes
- Restore the file from backup if required
- Isolate the host if compromise is suspected
- Rotate credentials if authentication files are impacted

---

## Escalation Criteria
Escalate immediately if:
- /etc/passwd, /etc/shadow, or sudoers is modified
- Multiple integrity alerts occur in a short time window
- Other suspicious activity is present
